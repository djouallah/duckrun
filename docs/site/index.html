duckrun — dbt model DAGs

Generated dbt documentation for the integration dbt projects (DuckDB → Delta Lake via delta-rs).

  • aemo The AEMO dbt project built against live Microsoft Fabric OneLake (abfss://). Full catalog with column metadata over the real Delta tables. source: djouallah/dbt_fabric_python_delta
  • merge_spill The incremental-MERGE spill benchmark project (merge_spill_bench): a chain of merge / append / overwrite ops on a TPCH lineitem. Built for real on a tiny scale factor, so the catalog carries Delta stats (row counts / size / last-modified). The heavy 60M-row spill run is a separate release gate, not the docs build. source: djouallah/duckrun
  • coffee The coffee-shop scenario (coffee_shop): ingest two dimension CSVs over https, dedup the SCD2 product dim, generate an N-row fact partitioned by region, and a revenue mart. Built for real, so the catalog carries Delta stats (row counts / size / last-modified). ported from: JosueBogran/coffeeshopdatageneratorv2
  • sde_dbt_tutorial The josephmachado/simple_dbt_project port (sde_dbt_tutorial): raw application tables → bronze typing → a Delta-backed SCD2 snapshot of the customer dim → a merge-incremental clickstream fact → the orders_obt gold mart joined through the SCD2 validity window. Built for real, so the catalog carries Delta stats (row counts / size / last-modified). source: josephmachado/simple_dbt_project

pure SQL — the connection APInot a dbt project

Everything above is dbt. This one isn't: demo_taxi.py is a runnable showcase of duckrun.connect() that reads live NYC TLC Yellow-Taxi data (Parquet + zone CSV, ~3M rows/month) straight off https and lands it into Delta on OneLake using nothing but conn.sql(...) — pure SQL, no dbt, no Python builders (bar the concurrency finale). Each step leans on DuckDB SQL the dbt path never needs.

Actual run: 50,000-row sample of live NYC TLC Yellow-Taxi 2024-01 → local Delta, 2026-06-21 UTC — ✅ all operations correct

#Statement — pure SQL via conn.sqlWhat it shows

4… QUALIFY row_number() OVER (…) ≤ 3top-3 zones per borough — one window pass, no self-join
5PIVOT (…) ON payment USING sum(fare)payment × time-of-day cross-tab
6GROUP BY ROLLUP (borough, payment)subtotals + grand-total super-aggregates
7list(struct_pack(…) ORDER BY …)[1:3], quantile_cont(…, [.5,.9,.99])LIST/STRUCT nested types & array quantiles
8… ASOF JOIN surge_card ON pickup_ts ≥ valid_fromsnap each trip to the latest surge band before it

10DELETE … (literal keys) + INSERT …a SQL-only upsert — no MERGE needed

12DeltaTable.merge × 2 (the one Spark-API step)two concurrent writers, one snapshot → the stale one is refused with CommitFailedError

Step 12 is the lone exception to pure SQL: conn.sql doesn't accept MERGE yet (#4), so the concurrency clash uses the Spark DeltaTable.merge builder — proving snapshot isolation holds.
